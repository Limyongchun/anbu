[ANBU 홈 화면 상태 충돌 버그 수정 — 초정밀 바이브코딩 지시서]

지금 문제
- 부모 카드 안에서 상태가 서로 다르게 표시되고 있다.
- 예:
  - DEBUG = SAFE
  - 본문 = 정상 활동 중
  - 배지 = 연결 끊김
- 이건 old 상태값과 new 상태값이 섞여 있기 때문이다.
- 반드시 “카드 전체”가 하나의 최종 상태만 사용하도록 통일해야 한다.

이번 작업 목표
- 부모 카드의 모든 상태 표시를 하나의 finalStatus 기준으로 통일한다.
- 상단 “확인 필요 N명” 집계도 같은 finalStatus 기준으로 다시 계산한다.
- 하트 클릭 시 lastAppActivityAt만 바뀌는 문제가 있으면, 연결 관련 시간도 함께 갱신한다.
- 이번 작업에서는 디자인 수정하지 말고 상태 충돌만 해결한다.

절대 하지 말 것
- 기존 구조를 크게 갈아엎지 마라.
- 새 폴더를 많이 만들지 마라.
- 푸시 알림 건드리지 마라.
- DB 구조 바꾸지 마라.
- 장소 로직 추가하지 마라.
- 카드 디자인 손대지 마라.
- 이번 작업은 “상태 소스 통일”만 한다.

-----------------------------------
1단계 — 현재 카드에서 상태가 표시되는 위치 전부 찾기
-----------------------------------

부모 카드 컴포넌트에서 아래 항목을 모두 찾는다.

1. 상태 배지 텍스트
   예: 연결 끊김 / 정상 / 확인 필요

2. 상태 배지 색상

3. 본문 상태 문구
   예: 정상 활동 중 / 기기 연결 상태를 확인해주세요

4. 카드 우측 액션 버튼 우선순위
   예: 전화 / 와이파이 / 위치 / 메시지

5. DEBUG 줄에 찍히는 상태

6. 상단 “N명 확인 필요” 카운트

위 6개가 각각 어떤 변수로 표시되는지 먼저 찾는다.
특히 아래 같은 이름을 찾는다.
- connectionStatus
- isConnected
- parent.status
- activityStatus
- debugStatus
- statusResult.status
- confirmedStatus
- signalLost
- oldStatus

목표:
카드의 상태 관련 UI가 현재 어떤 변수들을 섞어 쓰고 있는지 찾는다.

-----------------------------------
2단계 — 카드 최종 상태 변수 하나만 만든다
-----------------------------------

부모 카드 컴포넌트 최상단에 finalStatus 관련 변수를 만든다.

규칙:
- 카드에서 보여줄 상태는 오직 finalStatus 하나만 사용한다.
- finalStatus는 new status 로직 결과를 우선 사용한다.
- new status 가 없을 때만 oldStatus fallback 허용

예시 구조:

const newStatus = statusResult?.confirmedStatus || statusResult?.status || null;
const fallbackOldStatus = oldStatus || connectionStatus || 'SAFE';
const finalStatus = newStatus || fallbackOldStatus;

const finalReason = statusResult?.reason || 'normal';
const finalSummaryText =
  statusResult?.summaryText ||
  getSummaryTextFromStatus(finalStatus);

주의:
- finalStatus를 만든 뒤에는 카드 UI에서 connectionStatus를 직접 쓰지 마라.
- connectionStatus는 fallback 용도로만 남겨라.
- finalStatus 변수명은 그대로 써라. 다른 이름 만들지 마라.

-----------------------------------
3단계 — 상태 표시용 매핑 함수 4개 만든다
-----------------------------------

부모 카드 파일 안이나 가까운 util 파일에 아래 함수 4개를 만든다.

1. getStatusLabel(status)
2. getStatusColor(status)
3. getStatusSummary(status)
4. getPrimaryActions(status)

반드시 switch 문으로 작성한다.
if 문 여러 개로 흩어 쓰지 마라.

상태 규칙:

SAFE
- label: 정상
- summary: 정상 활동 중
- color: green 계열
- actions: 위치 보기, 전화

CHECK
- label: 확인 필요
- summary: 조용한 시간이 길어지고 있어요
- color: yellow 계열
- actions: 위치 보기, 메시지, 전화

DANGER
- label: 위험
- summary: 확인이 필요해요
- color: red 계열
- actions: 전화, 위치 보기

CRITICAL
- label: 즉시 확인
- summary: 확인전화가 필요해요
- color: dark red 계열
- actions: 전화, 마지막 위치 보기

SIGNAL_LOST
- label: 연결 끊김
- summary: 기기 연결 상태를 확인해주세요
- color: gray 계열
- actions: 전화, 연결 확인

중요:
- “연결 끊김” 텍스트는 finalStatus === SIGNAL_LOST 일 때만 표시해라.
- SAFE 인데 연결 끊김 배지를 보여주면 안 된다.

-----------------------------------
4단계 — 카드 UI 전체를 finalStatus만 보게 바꾼다
-----------------------------------

이제 부모 카드 안에서 아래 항목을 전부 finalStatus 기준으로 교체한다.

1. 상태 배지 텍스트
기존:
connectionStatus / oldStatus / isConnected 등 사용 금지
변경:
getStatusLabel(finalStatus)

2. 상태 배지 색상
기존:
연결상태용 별도 색상 사용 금지
변경:
getStatusColor(finalStatus)

3. 본문 상태 문구
기존:
connectionStatus 기반 문구 금지
변경:
finalSummaryText 또는 getStatusSummary(finalStatus)

4. 카드 우측 버튼
기존:
isConnected / signalLost / normal 등 개별 분기 금지
변경:
getPrimaryActions(finalStatus)

5. DEBUG 줄
기존:
statusResult.status만 따로 표시 가능
변경:
DEBUG에 아래 3개를 같이 보이게 해라
- computed: statusResult?.status
- confirmed: statusResult?.confirmedStatus
- final: finalStatus

이렇게 해야 충돌을 바로 확인 가능하다.

예시:
DEBUG 용천폰폰 | computed SAFE | confirmed SAFE | final SAFE

-----------------------------------
5단계 — 상단 “확인 필요 N명” 집계도 finalStatus 기준으로 바꾼다
-----------------------------------

지금 상단 숫자도 old/new 상태가 섞였을 가능성이 높다.
반드시 카드와 같은 기준으로 재계산해야 한다.

규칙:
- SAFE 는 확인 필요 인원에 포함하지 않는다.
- CHECK, DANGER, CRITICAL, SIGNAL_LOST 는 포함한다.

예시:

const needsAttentionStatuses = ['CHECK', 'DANGER', 'CRITICAL', 'SIGNAL_LOST'];

const attentionCount = parents.filter((parent) => {
  const statusResult = evaluateParentStatus(...);
  const finalStatus =
    statusResult?.confirmedStatus ||
    statusResult?.status ||
    parent.oldStatus ||
    'SAFE';

  return needsAttentionStatuses.includes(finalStatus);
}).length;

중요:
- 카드에 표시하는 상태 기준과 상단 집계 기준이 완전히 같아야 한다.
- 별도 connectionStatus 카운트 금지.

-----------------------------------
6단계 — 하트 버튼 클릭 시 활동만 말고 연결 시간도 함께 갱신
-----------------------------------

지금 사용자가 하트를 눌렀는데도 연결 끊김이 남아 있으면
하트 액션이 lastAppActivityAt만 바꾸고 있을 가능성이 크다.

하트 버튼 또는 부모 활동 ping 처리 함수에서 아래 값을 같이 갱신해라.

반드시 함께 갱신할 값:
- lastAppActivityAt = now
- lastHeartbeatAt = now
- isOnline = true
- lastConnectionHealthAt = now

가능하면 같이 갱신:
- lastSignalAt = now

주의:
- 하트는 단순 UI 효과가 아니라 “부모 기기가 살아있다”는 신호로 취급해라.
- 하트 클릭 후 1분 전 활동인데 SIGNAL_LOST가 남아 있으면 안 된다.

-----------------------------------
7단계 — SIGNAL_LOST 판정 완충 조건 추가
-----------------------------------

현재처럼 하트를 방금 눌렀는데 연결 끊김이 뜨는 문제를 막기 위해,
SIGNAL_LOST 판정 전에 완충 조건을 넣는다.

규칙:
- 최근 10분 이내 lastAppActivityAt 이 있으면 SIGNAL_LOST 바로 띄우지 마라.
- 최근 10분 이내 lastHeartbeatAt 이 있으면 SIGNAL_LOST 바로 띄우지 마라.
- 둘 중 하나라도 최근이면 최소 SAFE 또는 CHECK로 완화해라.

evaluateParentStatus 안에 아래 보정 규칙을 추가한다.

의사 규칙:
if (
  final candidate status is SIGNAL_LOST
  and (
    lastAppActivityAt within 10 minutes
    or lastHeartbeatAt within 10 minutes
  )
) {
  do not return SIGNAL_LOST;
  return SAFE or CHECK depending on inactivity;
}

중요:
- 위치 수신이 약간 늦는다고 바로 SIGNAL_LOST 하지 마라.
- 위치 + heartbeat + online 상태를 함께 보고 판단해라.

-----------------------------------
8단계 — 디버그 표시를 더 정확히 바꾼다
-----------------------------------

디버그 줄은 지금 상태를 눈으로 검증하는 용도이므로 아래처럼 바꾼다.

표시 항목:
- computed
- confirmed
- final
- lastAppActivityAt
- lastHeartbeatAt
- isOnline

예시:
DEBUG 용천폰폰
computed=SAFE
confirmed=SAFE
final=SAFE
app=1m
heartbeat=1m
online=true

예시:
DEBUG 황인철
computed=SIGNAL_LOST
confirmed=SIGNAL_LOST
final=SIGNAL_LOST
app=1980m
heartbeat=1980m
online=false

주의:
- DEBUG 한 줄이 너무 길면 2줄로 나눠라.
- production 배포에서는 숨겨라.

-----------------------------------
9단계 — 바로 검증할 테스트 시나리오
-----------------------------------

반드시 아래 순서로 직접 테스트한다.

테스트 A
- 하트 누르기
- 기대 결과:
  - finalStatus = SAFE
  - 배지 = 정상
  - 본문 = 정상 활동 중
  - 상단 확인 필요 인원 감소 또는 유지
  - 연결 끊김 배지 사라짐

테스트 B
- 위치 업데이트 끊기
- heartbeat도 끊기
- 오랜 시간 경과 mock
- 기대 결과:
  - finalStatus = SIGNAL_LOST
  - 배지 = 연결 끊김
  - 본문 = 기기 연결 상태를 확인해주세요

테스트 C
- lastAppActivityAt 4시간 전
- 위치/heartbeat는 최근
- 기대 결과:
  - finalStatus = CHECK
  - 배지 = 확인 필요
  - 본문 = 조용한 시간이 길어지고 있어요

테스트 D
- lastAppActivityAt 7시간 전
- 기대 결과:
  - finalStatus = DANGER

테스트 E
- lastAppActivityAt 13시간 전
- 기대 결과:
  - finalStatus = CRITICAL

가장 중요한 정상 기준:
용천폰폰처럼 하트 클릭 후 1분 전 활동이면
- DEBUG SAFE
- 본문 정상 활동 중
- 배지 정상
이 3개가 반드시 일치해야 한다.

-----------------------------------
10단계 — 작업 완료 기준
-----------------------------------

아래 조건을 모두 만족하면 이번 수정은 성공이다.

1. 카드의 배지 / 본문 / 버튼 / DEBUG 가 모두 finalStatus 기준으로 일치한다.
2. SAFE 인데 연결 끊김 배지가 나오지 않는다.
3. 하트 클릭 후 10분 이내에는 SIGNAL_LOST가 뜨지 않는다.
4. 상단 “확인 필요 N명” 숫자가 카드 상태와 맞는다.
5. old connectionStatus 값이 직접 UI에 표시되지 않는다.
6. red screen error 없이 동작한다.

-----------------------------------
11단계 — 마지막 안전장치
-----------------------------------

문제가 생기면 아래 한 줄만 바꿔서 즉시 롤백 가능하게 해라.

const USE_NEW_STATUS_UI = true;

규칙:
- true면 finalStatus 사용
- false면 기존 oldStatus 사용

즉, 모든 UI 연결은 아래 구조를 따라라.

const displayStatus = USE_NEW_STATUS_UI
  ? finalStatus
  : fallbackOldStatus;

이렇게 해야 문제가 생겨도 앱 전체가 죽지 않는다.

이번 작업의 핵심 한 줄 요약
- “카드 전체와 상단 집계를 하나의 finalStatus로 통일하고, 하트 클릭 시 연결 신호까지 함께 갱신해라.”
