ANBU 상태 변경 웨이팅 시간 구조

목표:
상태가 너무 자주 흔들리지 않게 하고,
자녀가 신뢰할 수 있는 홈 상태 문구와 알림 구조를 만든다.

핵심 원칙:
1. 데이터 기록은 즉시 한다
2. 상태 반영은 짧게 웨이팅한다
3. 위험 상태는 단계적으로 승급한다
4. 위치는 GPS 튐 방지를 위해 더 길게 웨이팅한다

적용 기준:

1) 활동 감지
- 사진 보기, 앱 터치, 버튼 클릭, 수동 체크인은 즉시 lastActiveAt 갱신
- 단, 홈 상태 문구는 2분 후 반영
- 최근 활동 있음 상태는 마지막 활동 후 2시간까지 유지

2) 위치 상태
- 저장된 장소(집, 노인정, 병원, 친구집) 진입 감지 시 즉시 후보 상태 저장
- 3분 이상 같은 장소에 머물면 해당 위치로 확정
- 저장된 장소에서 벗어난 뒤 3분 이상 유지되면 이탈 확정
- 이동중 상태는 2분 이상 연속 이동 패턴일 때 확정
- 먼 거리 새 장소는 5분 이상 유지 시 확정

3) 확인 필요 / 위험 상태
- 정상 → 확인 필요: 기준 시간 초과 후 20분 뒤 확정
- 확인 필요 → 위험: 추가 40분 뒤 확정
- 즉, 위험 상태는 바로 보내지 말고 최소 2단계로 올린다

4) 취침 시간 로직
- 자녀가 취침 시간 / 기상 시간을 설정 가능하게 한다
- 취침 시작 판정은 설정 취침 시간 도달 후 30분 지나서 적용
- 기상 확인 필요는 설정 기상 시간 후 1시간 동안 활동 없을 때 적용
- 기상 위험은 그 이후 추가 1시간 이상 무활동일 때 적용

권장 기본값:
- activityUiDelay = 2분
- placeEnterDelay = 3분
- placeExitDelay = 3분
- movingDelay = 2분
- farPlaceConfirmDelay = 5분
- checkNeedDelay = 20분
- dangerDelay = 40분
- sleepApplyDelay = 30분
- wakeCheckDelay = 60분

상태 승급 구조:
1. 정상
2. 확인 필요 후보
3. 확인 필요
4. 위험 후보
5. 위험

홈 화면 문구 예시:
- 집에 계세요
- 노인정에 계세요
- 이동 중이에요
- 편안히 쉬고 계세요
- 조금 더 지켜보는 중이에요
- 확인이 필요해요
- 장시간 변화가 없어 확인이 필요합니다

개발 규칙:
- 이벤트 발생 즉시 DB에 로그 저장
- 상태 계산기는 후보 상태를 먼저 계산
- 웨이팅 시간 통과 후에만 확정 상태 변경
- 확정 상태가 바뀔 때만 홈 UI 갱신
- 위험 알림은 확정 상태에서만 발송
- 후보 상태에서는 푸시 알림 금지

상태 계산 규칙 요약:

A. 활동
- 활동 이벤트가 들어오면 lastActiveAt 즉시 갱신
- 단, UI는 2분 뒤 반영

B. 위치
- 새 위치 감지되면 candidatePlace 저장
- 3분 유지되면 currentPlace 확정
- GPS가 흔들리면 후보만 바뀌고 확정은 안 바뀜

C. 시간
- 취침 시간 직후 바로 취침 상태로 바꾸지 않고 30분 뒤 적용
- 기상 시간 이후 1시간 무활동이면 확인 필요
- 추가 1시간 이상 무활동이면 위험

D. 위험도
- 기준 초과 즉시 CHECK 아님
- 먼저 CHECK_CANDIDATE
- 20분 유지되면 CHECK
- 그 후 40분 더 지나면 DANGER

알림 규칙:
- SAFE: 알림 없음
- CHECK_CANDIDATE: 알림 없음
- CHECK: 1회 알림
- DANGER_CANDIDATE: 알림 없음
- DANGER: 반복 알림 또는 강조 카드 표시

절대 규칙:
- 후보 상태에서는 절대 푸시 알림 보내지 않는다
- 상태 확정이 바뀔 때만 알림을 보낸다
- 위치는 즉시 바꾸지 않는다
- 활동 로그는 즉시 저장하지만 메시지는 바로 바꾸지 않는다

바이브코딩 작업 지시:

1. statusConfig.ts를 만들어 웨이팅 시간을 상수로 관리한다
2. statusEngine.ts에서 후보 상태와 확정 상태를 분리한다
3. 홈 화면은 확정 상태만 보여준다
4. 위치는 candidatePlace 방식으로 2단계 확정 구조를 사용한다
5. CHECK, DANGER는 단계 승급 구조로 만든다
6. 푸시 알림은 CHECK, DANGER 확정 상태에서만 발송한다
7. 상태 변경 이력은 로그 테이블에 저장한다
8. 활동 이벤트는 즉시 저장하고 UI는 2분 디바운스로 갱신한다

최종 한 줄 원칙:
즉시 기록 + 지연 반영 + 후보 상태 + 확정 상태 + 단계적 알림
